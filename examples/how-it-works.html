How Playpad works

A tour of the architecture behind a stateless, end-to-end encrypted, zero-server pastebin for AI-coding playgrounds.

Stateless E2E encrypted $0 infra No DB · No accounts

1 · The trick: URL fragments never reach the server

RFC 3986 §3.5 defines the fragment as the part of a URL after the #. It is a client-only locator. Browsers do not include it in HTTP requests. Servers do not see it, log it, or know it exists.

https://playpad.dev/#d=eJxdj81qwzAQhF9F7DmF3JLcCk1aSqGl0EsP...

Playpad puts the entire bundle in that fragment. The static page loads from any host, then JavaScript reads window.location.hash locally and renders.

Why this matters: the host literally cannot read your content. Not "we promise we don't" — there is no inbound data to read.

2 · Packing the bundle

The playground (title + files + metadata) becomes JSON, then walks through this pipeline:

STEP 1
JSON
The bundle structure: title, files, optional note.
STEP 2
deflate (raw)
pako, level 9. Typically 60–75% smaller for text.
STEP 3
base64url
URL-safe alphabet, no padding. Drops in directly into #.
STEP 4
Prefix
d= for plaintext, e= for encrypted.

3 · Optional encryption (E2E)

Set a password and the bundle is encrypted entirely in your browser before going into the URL.

  • Key derivation: PBKDF2-SHA256, 600,000 iterations (OWASP 2023+ recommendation), 16-byte random salt
  • Cipher: AES-GCM-256, 12-byte random IV per encryption
  • Authentication: GCM is AEAD — wrong password or tampered ciphertext is rejected without a decryption oracle
  • Format: [salt 16B][iv 12B][ciphertext], then base64url. No JSON wrapper, no metadata leaks
  • Crypto library: the browser's Web Crypto API. No third-party crypto code in the bundle

4 · Rendering arbitrary HTML, safely

If the bundle contains an .html file, Playpad renders it in a sandboxed iframe:

<iframe srcdoc="..." sandbox="allow-scripts allow-modals" referrerpolicy="no-referrer">
  • No allow-same-origin → iframe gets a null origin, can't touch Playpad's storage or session
  • No allow-top-navigation → can't redirect the parent
  • No allow-forms, no allow-popups, no allow-downloads
  • JS still runs (you can render Mermaid, do animations) — just isolated from the host page

5 · Trade-offs

  • URLs are long. ~8K to 30K chars typical. Caps at 60K (browser-safe). Past 8K, some chat clients truncate
  • No revocation. The fragment is the data; if the link leaked, the password is the only protection
  • No expiration. No server means no clock to enforce one
  • No analytics, no ACLs, no view counts. By design

6 · Why this is feasible in 2026

  • Web Crypto API gives you PBKDF2 + AES-GCM in every browser, no shims
  • Modern browsers tolerate 64K+ URLs without issue
  • Free static hosting with edge CDN (Cloudflare Pages, Vercel) is effectively unlimited
  • Total bundle: ~52 KB gzipped (React + Vite, pako, all UI). Loads instantly
